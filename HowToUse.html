Addresser tool

Assigner tool

This mostly-javascript “program” (more like a script) allows you to manage your RAM usage easily. It came to my attention that some beginners or anyone unaware of how memory usage works complains that one patch breaks another often was caused by RAM conflicts when installing ASM resources (patch, blocks, sprites, uberasm, etc.). Here is an example (scroll and find DTA450's comment). He probably either forgot to change the freeram define setting OR assume that no RAM would take that address, and BytesConsumed-1 consecutive bytes after it. In this case, the Directional quake and the wall kick patch have issues with RAM $61 when using default RAM addresses.

Note: To easy-copy text inside code blocks here, hold CTRL and double-click the text. Same applies in other HTMLs in this package.

There are two tools:

  • The “Addresser”, which takes a given RAM range (either a literal range, or a starting byte address with a length/number-of-bytes given) and checks for overlaps and gaps in a sequential address order. It has a visualizer (“Byte view”) and an option to offset the RAM usage to another address if you are dealing with mirroring and SA-1 remapping.
  • The “Assigner”, which takes a given list of ASM defines and a list of empty RAMs and automatically assigns the ASM defines to every currently-unused RAM address.

This is especially very useful for collaboration hacks or any hacks that uses a ton of ASM resources, as manually pinpointing a problem can be difficult.

What you need to do is keep track of what RAM is being used, both by SMW and the ASM resources. Make sure they formatted as stated in those HTML javascript files.

To use addresser, just copy the list in SMW_UsedAddress.txt and paste it in the HTML's textarea, it will output a list and a table showing their statuses as well as discovered gaps EDIT: It automatically fills in the data when you open the HTML file. You can also view them by Byte view, which will display a graph of every bytes in the sequence on how they're occupied.

To add to above, you CAN add your own used RAMs here by the ASM resource you installed.

To use the auto-assigner:

  1. Have any ASM resource that consumes freeram. For example, this wall kick patch and the directional quake mentioned earlier: 
    !flags			= $61		; The wallkick flags. (RAM)
!no_back_timer		= $62		; The timer for not moving back. (RAM)
!temp_y_spd		= $63		; The temporary Y speed. (RAM)

     if !sa1 == 0
  !Freeram_DirQuake_Layer1XDisplacement		= $60
 else
  !Freeram_DirQuake_Layer1XDisplacement		= $60
 endif
  ;^[2 bytes], same as $1888 but for X position displacement. Without this
  ; displacement ram address, sprite positions will appear also "displaced", not
  ; syncing with layer 1 (like appearing that their foot goes partially through the floor).
    Now, not all ASM resources would label how much bytes they require. Normally this takes 1 or 2 bytes. If you have some ASM knowledge, when RAM is loaded in 16-bit, then it is 2 bytes (assuming it doesn't zero out the high byte). If it had an index attached to it like this: LDA !Freeram,x, then it could take X+1 bytes where X is an index value at the highest possible value. In this example, wall kick patch all take 1 byte each, and the directional quake takes 2 bytes.

  2. Next, open the autoassigner and write your RAM usage of the ASM resource you're dealing with into the first textarea. Make sure it's formated as stated at the top, and make sure you supply the FULL address and not the mirroring address (see the notes at the bottom of why). The ASM resource name isn't really much processed other than it has to exist (at least 1 character) and is used for sorting the display, so any name without the tab character would work, I just recommend making sure you name them in a way it is easy to locate and such. Here is using the previous example: 
    WallKick	!flags	1	-	-
WallKick	!no_back_timer	1	-	-
WallKick	!temp_y_spd	1	-	-
DirectionalQuake	!Freeram_DirQuake_Layer1XDisplacement	2	-	-
    By the way, you can use space instead of tabs.

  3. Now obtain a list of empty RAMs. You can do this by opening SMW_UsedAddress.txt, copy the list, paste it in the addresser EDIT: It's auto-filled, click the “Quick-copy gaps” button, and paste it in the second textarea of assigner.

  4. The assigner will now output the assigned RAMs at the bottom, asar ASM formatted by default. I highly recommend having it sorted by resource name (which is set to that by default) so you can easily grab all the defines relating to a given ASM resource (such as to avoid placing a define from one resource into another). You can then easily set the RAMs in those ASM files by commenting out the original defines you are dealing with and then paste the results from the assigner tool below it (and before any code that uses the defines), like this: 
    ;Wallkick patch
;!flags			= $61		; The wallkick flags. (RAM)
;!no_back_timer		= $62		; The timer for not moving back. (RAM)
;!temp_y_spd		= $63		; The temporary Y speed. (RAM)

!flags = $7E0058	; [1 byte] WallKick
!no_back_timer = $7E005C	; [1 byte] WallKick
!temp_y_spd = $7E0060	; [1 byte] WallKick
    ;directional quake
; if !sa1 == 0
;  !Freeram_DirQuake_Layer1XDisplacement		= $60
; else
;  !Freeram_DirQuake_Layer1XDisplacement		= $60
; endif
!Freeram_DirQuake_Layer1XDisplacement = $7E0061	; [2 bytes] DirectionalQuake

    Don't forget to “shorten” your address when possible (replace $7E005C with $5C, $7E0060 with $60 and so on) (add semicolons “;” before the string to be read) (easier using notepad++ shift+alt to make your selections column mode) and paste the assigned defines you got from the assigner below it. Don't forget you can also add that to your list of Used RAM by clicking on “Address tracker format”, copy that format, and paste it in your list of used RAM, so that any future ASM consuming RAM will avoid conflicting with that updated list.

Notes

  • The freeram defines in the first textarea of autoassigner are processed from longest to shortest number of bytes, and the second textarea are processed from lowest to highest address. This means the longest RAM define will get assigned to the lowest available address (accounting the min and max address as well as the size of it).
  • This entire tool does not account to bank borders, so do not assume things like a range of $7EFFFF-$7F0000 that it avoids it. It also does not account for RAM mirroring, so make sure you place your full address in the tool.
    • Also, the assigner tool does not merge RAM ranges that are touching each other, so any define will not have its length cross that boundary. For example: A range from $7E0060 to $7E0061 and another ranging from $7E0062 to $7E0063, that is 2 ranges each having 2 bytes long, does not allow a 3 or 4 -bytes long freeram define to be assigned anywhere here.
  • This tool can be used beyond smw modding, since this just checks address ranges and nothing more.
  • Some RAM may be reused for a different purpose when not used for a given specific purpose, such as reusing $60 for a jump counter during a level, and for something else during overworld. If this is the case then I would recommend having a list of used RAM being separated from “global” uses into “local” uses, it is only a problem if 2+ purposes are used at the same time. The same applies for custom scratch RAM besides $00-$0F.
  • When using the auto-assigner tool, be careful of data fragmentation when defines get assigned to an address in a way that results in bytes currently free surrounding the now-used bytes. This can happen if you set the minimum and maximum address forcing it to be placed so that its starting address is not at the starting address of the currently empty RAM range.

Useful tips

  • It's possible to make asar output a txt file in the format the tools here would accept, here's how to do it
    1. Create an ASM defines file (example, DefineTest.asm), or have it in the ASM file with the code doing this: 
      !Freeram_test = $7E0060	;>Lets assume this takes 4 bytes.
!Size	= 4		;>Some patches or any other ASM resource may have ram with a certain size depending on the settings.

print "$", hex(!Freeram_test), "	", dec(!Size), "	Description here"
    2. Then create a batch file (test.bat), in that file to contain this text (each space between the indicated brackets is a parameter separator, and a newline submits the data): 
      [path to asar.exe] [path to game file] > [path location to create the txt file]
pause
    3. If there are spaces in the file path, wrap the whole path in quotes. Then save, run it (should open the command prompt). Example: 
      "C:\Users\GreenHammerBro\Desktop\smw romhacking\General Tools\asar181\asar.exe" "C:\Users\GreenHammerBro\Desktop\smw romhacking\Submission\AddresserTool\DefineTest.asm" "C:\Users\GreenHammerBro\Desktop\smw romhacking\Submission\AddresserTool\a.smc" >> "C:\Users\RedBro\Desktop\smw romhacking\Submission\AddresserTool\Output.txt"
pause
    4. And a file should appear with the RAM info (at [path location to create the txt file]). Note: this HAS to be outputted as a text file, the command prompt will convert it into 5 regular space characters when copied directly from the console window.
